# Requirements Document

## Introduction

This document specifies requirements for enhancing the kiro-pants-power output capture system to provide structured data extraction and improved error diagnostics for AI agents. The current implementation captures raw stdout/stderr but lacks parsing capabilities for structured formats (JUnit XML, coverage reports, MyPy output) and detailed error context. This refactor will enable agents to receive actionable error information, test failure details, and access to generated artifacts.

## Glossary

- **Output_Capture_System**: The component responsible for capturing, parsing, and formatting output from Pants build system commands
- **JUnit_XML_Parser**: Component that parses JUnit XML test report files into structured data
- **Coverage_Report_Parser**: Component that parses coverage reports in JSON or XML format
- **MyPy_Output_Parser**: Component that extracts type checking errors and statistics from MyPy output
- **Sandbox_Path_Extractor**: Component that extracts sandbox directory paths from Pants log output
- **Error_Diagnostic_Formatter**: Component that formats detailed error information for agent consumption
- **Structured_Output**: Machine-readable data format (JSON, XML) containing parsed results
- **Agent**: AI system consuming the power's output to make development decisions
- **Pants**: The build system being wrapped by the kiro-pants-power
- **Test_Report**: JUnit XML file generated by Pants containing test execution results
- **Coverage_Report**: JSON or XML file containing code coverage metrics
- **Sandbox**: Hermetic temporary directory where Pants executes processes
- **Pretty_Printer**: Component that formats Configuration objects back into valid configuration files

## Requirements

### Requirement 1: Parse JUnit XML Test Reports

**User Story:** As an agent, I want structured test result data, so that I can identify which specific tests failed and why without parsing raw console output.

#### Acceptance Criteria

1. WHEN a test command completes, THE Output_Capture_System SHALL check for JUnit XML reports in the configured output directory
2. WHEN a JUnit XML report exists, THE JUnit_XML_Parser SHALL parse the XML file into structured data
3. THE JUnit_XML_Parser SHALL extract test case names, status (passed/failed/skipped), execution time, and failure messages
4. WHEN test failures exist, THE Error_Diagnostic_Formatter SHALL include the test name, failure type, failure message, and stack trace location
5. WHEN multiple test files execute, THE JUnit_XML_Parser SHALL aggregate results across all report files
6. THE Output_Capture_System SHALL return structured test results containing total count, pass count, fail count, skip count, and failure details

### Requirement 2: Parse Coverage Reports

**User Story:** As an agent, I want coverage metrics in structured format, so that I can assess test coverage without manual report inspection.

#### Acceptance Criteria

1. WHEN a test command executes with coverage enabled, THE Output_Capture_System SHALL check for coverage reports in the configured output directory
2. WHERE coverage reports exist in JSON format, THE Coverage_Report_Parser SHALL parse JSON coverage data
3. WHERE coverage reports exist in XML format, THE Coverage_Report_Parser SHALL parse XML coverage data
4. THE Coverage_Report_Parser SHALL extract overall coverage percentage, per-file coverage, and uncovered line ranges
5. THE Output_Capture_System SHALL return structured coverage data containing total coverage percentage and per-file metrics
6. WHEN coverage data is unavailable, THE Output_Capture_System SHALL indicate coverage was not captured without failing

### Requirement 3: Extract MyPy Type Checking Errors

**User Story:** As an agent, I want detailed type checking errors, so that I can understand what type issues exist without running commands manually.

#### Acceptance Criteria

1. WHEN a check command fails with MyPy errors, THE MyPy_Output_Parser SHALL extract error details from console output
2. THE MyPy_Output_Parser SHALL parse each error to extract file path, line number, error code, and error message
3. THE Error_Diagnostic_Formatter SHALL format type errors with file location, line number, and full error description
4. WHEN multiple files have type errors, THE MyPy_Output_Parser SHALL aggregate all errors by file
5. THE Output_Capture_System SHALL return structured type error data containing error count and detailed error list
6. WHEN MyPy generates report files, THE Output_Capture_System SHALL preserve report file paths for agent access

### Requirement 4: Extract Sandbox Paths for Artifact Inspection

**User Story:** As an agent, I want sandbox directory paths when processes fail, so that I can inspect artifacts and reproduce failures.

#### Acceptance Criteria

1. WHEN Pants executes with keep-sandboxes enabled, THE Sandbox_Path_Extractor SHALL parse log output for sandbox preservation messages
2. THE Sandbox_Path_Extractor SHALL extract the full sandbox directory path from log messages
3. THE Sandbox_Path_Extractor SHALL associate sandbox paths with the corresponding process description
4. WHEN a process fails, THE Output_Capture_System SHALL include the sandbox path in error output
5. THE Output_Capture_System SHALL return sandbox paths for all preserved sandboxes
6. WHEN no sandboxes are preserved, THE Output_Capture_System SHALL indicate sandboxes were not kept without failing

### Requirement 5: Configure Pants for Structured Output Generation

**User Story:** As a power administrator, I want Pants configured to generate structured reports, so that parsing capabilities can function correctly.

#### Acceptance Criteria

1. THE Output_Capture_System SHALL configure test commands with `--test-report` flag to enable JUnit XML generation
2. THE Output_Capture_System SHALL configure test commands with `--test-use-coverage` flag when coverage is requested
3. THE Output_Capture_System SHALL configure coverage output format to include JSON or XML via environment or config
4. THE Output_Capture_System SHALL configure test output mode to `failed` to reduce noise while preserving error details
5. WHERE MyPy report generation is configured, THE Output_Capture_System SHALL preserve MyPy report directory paths
6. THE Output_Capture_System SHALL configure `--keep-sandboxes=on_failure` for failed process inspection

### Requirement 6: Format Enhanced Error Diagnostics

**User Story:** As an agent, I want concise, actionable error summaries, so that I can quickly understand what failed without reading full output dumps.

#### Acceptance Criteria

1. WHEN a command fails, THE Error_Diagnostic_Formatter SHALL create a structured error summary
2. THE Error_Diagnostic_Formatter SHALL include error type (test failure, type error, execution error)
3. WHEN test failures occur, THE Error_Diagnostic_Formatter SHALL list failed test names with failure reasons
4. WHEN type errors occur, THE Error_Diagnostic_Formatter SHALL list files with error counts and sample errors
5. THE Error_Diagnostic_Formatter SHALL limit error detail output to the most relevant information (first N errors per category)
6. THE Error_Diagnostic_Formatter SHALL include sandbox paths when available for detailed investigation

### Requirement 7: Handle Parsing Errors Gracefully

**User Story:** As an agent, I want the power to handle parsing failures gracefully, so that I still receive useful output even when structured parsing fails.

#### Acceptance Criteria

1. WHEN XML parsing fails, THE Output_Capture_System SHALL log the parsing error and return raw output
2. WHEN report files are missing, THE Output_Capture_System SHALL indicate reports were not found without failing the command
3. WHEN report files are malformed, THE Output_Capture_System SHALL attempt partial parsing and indicate which sections failed
4. IF a parser encounters an unexpected format, THEN THE Output_Capture_System SHALL fall back to raw output capture
5. THE Output_Capture_System SHALL never fail a command solely due to parsing errors
6. THE Output_Capture_System SHALL include parsing error details in verbose mode for debugging

### Requirement 8: Support Streaming Progress for Long Operations

**User Story:** As an agent, I want real-time progress updates during long-running commands, so that I can monitor execution status.

#### Acceptance Criteria

1. WHEN a command executes, THE Output_Capture_System SHALL stream console output in real-time
2. THE Output_Capture_System SHALL preserve output ordering (stdout and stderr interleaved correctly)
3. WHEN streaming is enabled, THE Output_Capture_System SHALL buffer output for both real-time display and final parsing
4. THE Output_Capture_System SHALL indicate command progress through incremental output updates
5. WHEN a command completes, THE Output_Capture_System SHALL perform final parsing on complete output
6. THE Output_Capture_System SHALL support both streaming and buffered modes based on configuration

### Requirement 9: Provide Workflow Step Progress in full_quality_check

**User Story:** As an agent, I want to see which quality check steps passed before a failure, so that I understand where in the pipeline the issue occurred.

#### Acceptance Criteria

1. WHEN full_quality_check executes, THE Output_Capture_System SHALL track completion status of each step (fix, lint, check, test)
2. WHEN a step completes successfully, THE Output_Capture_System SHALL indicate step completion with step number and name
3. WHEN a step fails, THE Output_Capture_System SHALL indicate which step failed and include enhanced error diagnostics
4. THE Output_Capture_System SHALL provide a workflow summary showing completed steps and failure point
5. THE Output_Capture_System SHALL include timing information for each completed step
6. WHEN the workflow completes successfully, THE Output_Capture_System SHALL show all steps passed with total execution time

### Requirement 10: Parse and Format Pytest Output

**User Story:** As an agent, I want detailed pytest failure information, so that I can understand assertion failures and error context.

#### Acceptance Criteria

1. WHEN pytest output contains failure sections, THE Output_Capture_System SHALL extract the failure summary section
2. THE Output_Capture_System SHALL parse pytest short test summary info (FAILED lines)
3. THE Output_Capture_System SHALL extract assertion failure details including expected vs actual values
4. WHEN pytest shows stack traces, THE Output_Capture_System SHALL include relevant stack trace frames (not full trace)
5. THE Output_Capture_System SHALL identify test failure types (AssertionError, Exception, etc.)
6. THE Output_Capture_System SHALL format pytest failures in a structured format with test name, failure type, and message

### Requirement 11: Configuration File Parsing and Pretty Printing

**User Story:** As a developer, I want to parse and format Pants configuration files, so that I can programmatically modify build settings.

#### Acceptance Criteria

1. WHEN a Pants configuration file (pants.toml) is provided, THE Output_Capture_System SHALL parse it into a structured Configuration object
2. WHEN an invalid configuration file is provided, THE Output_Capture_System SHALL return a descriptive error indicating the syntax issue and location
3. THE Pretty_Printer SHALL format Configuration objects back into valid TOML configuration files
4. FOR ALL valid Configuration objects, parsing then printing then parsing SHALL produce an equivalent object (round-trip property)
5. THE Pretty_Printer SHALL preserve comments and formatting where possible
6. THE Output_Capture_System SHALL validate configuration schema against known Pants configuration options
