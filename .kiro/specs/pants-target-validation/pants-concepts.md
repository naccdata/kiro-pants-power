# Pants Target Concepts - Research Notes

Source: [Pants Documentation - Targets and BUILD files](https://www.pantsbuild.org/dev/docs/using-pants/key-concepts/targets-and-build-files)

## Key Concepts

### What are Targets?

Targets are addressable sets of metadata describing code. They are defined in BUILD files and provide information Pants needs to build, test, lint, etc.

**Common target types:**
- First-party code: `python_source`, `python_test`, `shell_source`, `resource`, `file`
- Third-party: `python_requirement`
- Artifacts: `pex_binary`, `archive`
- Target generators: `python_tests` → `python_test`, `python_sources` → `python_source`

### Target Addresses

Targets are identified by addresses in the form: `path/to/dir:name`

**Address formats:**
- `path/to/dir:target_name` - Specific target in a directory
- `path/to/dir:` - Default target (when name matches directory)
- `//:target_name` - Target at repository root
- `:target_name` - Relative address (same directory)

### Target Generation

Many targets are "target generators" that create multiple targets:

**First-party generators:**
- `python_tests(sources=["test_*.py"])` generates one `python_test` per file
- Generated address: `path/to/test_file.py:generator_name`
- If generator uses default name: just `path/to/test_file.py`

**Third-party generators:**
- `go_mod` generates `go_third_party_package` targets
- Generated address: `path/to:generator#package_name`

**Key insight:** You can use the generator address as an alias for ALL generated targets.
- `pants test project:tests` runs all tests generated by `project:tests`
- `pants test project/test_file.py:tests` runs just that one test

### Special Address Syntax

**Wildcards:**
- `::` - All targets in directory and subdirectories (recursive)
- `path/to/dir::` - All targets under specific directory (recursive)
- `path/to/dir:` - All targets in specific directory (non-recursive)

**File paths:**
- `path/to/file.py` - Shorthand for the generated target from that file
- Only works if file is in a target generator's `sources` field

### Dependencies Field

The `dependencies` field lists other targets needed:
- Usually inferred automatically from imports
- Can add explicit dependencies: `dependencies=["assets:logo", "3rdparty/python:requests"]`
- Relative addresses: `:target_name` for same directory
- Exclude with `!`: `dependencies=["!3rdparty/python:numpy"]`

### Parametrization

Create multiple targets with different metadata using `parametrize()`:

```python
python_test(
    name="tests",
    source="test.py",
    interpreter_constraints=parametrize(
        py2=["==2.7.*"],
        py3=[">=3.6"]
    )
)
```

Creates addresses:
- `example:tests@interpreter_constraints=py2`
- `example:tests@interpreter_constraints=py3`

Can use `example:tests` as alias for both.

## Implications for Intent-Based API

### What Users Actually Want to Express

Based on Pants concepts, users want to:

1. **Run all tests everywhere**: `::` (all targets recursively)
2. **Run tests in a directory tree**: `path/to/dir::` (directory + subdirs)
3. **Run tests in one directory**: `path/to/dir:` (just that directory)
4. **Run a specific test target**: `path/to/dir:target_name`
5. **Run tests from a file**: `path/to/test_file.py` (generated target)
6. **Run a specific test function**: Need to use Pants' native filtering

### Key Insights

1. **File paths are already intent-based** - `path/to/test.py` is simpler than `path/to/test.py:tests`
2. **`::` is the most common pattern** - "run all tests in this area"
3. **Target generators are aliases** - `project:tests` already means "all tests"
4. **Pants has built-in test filtering** - Can filter by test name using Pants options

### Recommended Intent-Based Design

Instead of exposing raw target syntax, provide clear intent options:

```python
pants_test(
    # Scope of what to test
    scope="all" | "directory" | "file",
    
    # Path (required for directory/file scopes)
    path=None,
    
    # Whether to include subdirectories (for directory scope)
    recursive=True,
    
    # Filter to specific test names (uses Pants' -k option)
    test_filter=None,
    
    # Other Pants options...
)
```

**Mapping to Pants targets:**
- `scope="all"` → `::` 
- `scope="directory", path="src/tests", recursive=True` → `src/tests::`
- `scope="directory", path="src/tests", recursive=False` → `src/tests:`
- `scope="file", path="src/tests/test_foo.py"` → `src/tests/test_foo.py`
- `test_filter="test_create"` → adds `-k test_create` to Pants command

This abstracts away:
- Target generator names
- Generated target address syntax
- The difference between `:` and `::`
- BUILD file structure

While still giving full power through clear intent.
